Language Specification · Version 1.0.0
ANAL

Append-oriented, Narrow-access Language. A strictly sequential, stack-based programming language designed around one governing principle: everything enters from one end, in order, with consent.

stack-based strongly typed no pull requests consent enforced turing complete .anal · .sph
Note
This document describes ANAL v1.0. The current reference interpreter implements v0.2 — a strict subset. See the milestones for what's shipped and what's still spec'd.
01 Design Philosophy
Principle
"ANAL does not meet you halfway. ANAL expects you to come to it." — The ANAL Manifesto, §1

ANAL was designed around a single governing principle: everything enters from one end, in order, with consent. There are no shortcuts, no backdoors, no random access. The stack is sacred.

Unlike languages that permit arbitrary memory access, object mutation, or concurrent execution, ANAL enforces sequentiality as a virtue, not a limitation. If you need random access, you have chosen the wrong language. ANAL is not sorry.

Core Tenets

Append only, by default. The natural direction of ANAL is forward. Removal is possible but deliberate.

Consent before destruction. Any operation that permanently removes or modifies data at depth requires the CONSENT flag. This is not optional.

Preparation before penetration. Deep stack operations require PREP. Skipping it raises a TIGHTNESS error. The runtime does not apologise for this.

No pulling. ANAL does not support pull requests, pull-based I/O, or any metaphor involving pulling. Everything is pushed.

Sequential or nothing. ANAL has no threading model. Proposals to add one are closed without comment.

02 Syntax & Structure

ANAL programs are plain text files with the .anal extension. Each statement occupies one line. Comments begin with ;. ANAL is case-sensitive — all keywords are uppercase. Writing a keyword in lowercase raises a CASING error.

; This is a comment.

PUSH 42         ; push integer
PUSH "hello"     ; push string
PUSH 3.14       ; push float

loop_start:     ; label declaration
  DILATE        ; begin loop block
    EXPEL
  CONSTRICT     ; end loop block

File Structure

An ANAL program has three optional sections, in strict order: header, definitions, execution.

; 1. Header — metadata and imports
ANAL "my_program" VERSION 1
INGEST "stdlib/math"

; 2. Definitions — PASSAGES (functions)
PASSAGE square:
  DUP
  MUL
EXIT

; 3. Execution — the main body
PUSH 9
ENTER square
EXPEL    ; → 81
03 Stack Operations

The stack is the heart of ANAL. All data lives on the stack. All computation moves through the stack. There is only the stack.

Command Args Description
PUSH <value> Insert a value at the top of the stack. The primary method of input. ANAL strongly prefers PUSH to all alternatives.
Accepts INT, FLOAT, STRING, BOOL, BLOC. Does not accept nulls. ANAL has no concept of nothing.
POP Remove and return the top value. The value is gone.
Raises EMPTINESS if stack is empty. Raises LOCKDOWN if CLENCHed.
PROBE Inspect the top value without removal. Non-invasive. PROBE is the only truly safe operation in ANAL. always safe
May be called at any time, in any state, even during CLENCH.
INSERT <depth> <val> Inject a value at an arbitrary depth. Requires PREP immediately prior. needs PREP
Raises TIGHTNESS if PREP was not called. Raises PENETRATION_DEPTH if depth exceeds stack size.
EXTRACT <depth> Remove a value from an arbitrary depth. The gap collapses cleanly. needs CONSENT
Raises REFUSAL without CONSENT. Raises PENETRATION_DEPTH if out of bounds.
SWAP Exchange the top two values.
DUP Duplicate the top value and push the copy.
OVER Copy the second-from-top value to the top: (a b -- a b a).
The principal way to keep two values alive at once. Pairs with DUP and SWAP as the minimum useful stack vocabulary.
ROT Rotate the third-from-top value to the top: (a b c -- b c a).
Three applications of ROT return the stack to its original order. Use when a buried value must be brought forward without losing the values above it.
NIP Drop the second-from-top value: (a b -- b). Equivalent to SWAP POP but read-flat as a single op.
Common after OVER LOAD <i>: the duplicated handle is no longer needed once the value has been read.
DEPTH Push the current stack depth as an INT. Useful before INSERT or EXTRACT.
FLUSH Clear the entire stack. Cathartic. Permanent. needs CONSENT
Raises REFUSAL without prior CONSENT.
04 Control & State
Command Args Description
PREP Prepare the stack for deep operations. Sets internal readiness flag. Must immediately precede INSERT.
Readiness flag clears after one INSERT. Good hygiene. Always PREP.
CLENCH Freeze the stack. No write operations permitted until RELEASE. PROBE still works.
Nested CLENCHes are cumulative. Each requires a matching RELEASE.
RELEASE Unfreeze the stack. Decrements CLENCH depth by one.
Raises PREMATURE_RELEASE if called on an unclenched stack.
CONSENT Set the consent flag. Required before FLUSH, EXTRACT, or any destructive deep operation. mandatory
Clears automatically after one destructive operation. Must be re-given each time. ANAL does not assume ongoing consent.
RELAX Clear any armed PREP or CONSENT flag. always safe
Idempotent: a no-op if neither flag is armed. Allowed during CLENCH since it does not touch the stack. Consent withdrawn before use is consent never given. ANAL forgives forgetting.
EXPAND <n> Reserve capacity for n more values above the current depth. Call before you need it, not after.
Monotonic: subsequent EXPANDs only raise the cap, never lower it. A push past the cap raises OVERFLOW. Stacks with no EXPAND are uncapped and never overflow.
HOLD [ms] Pause execution. With an INT operand, sleeps that many milliseconds. Without one, waits for a RESUME signal.
Stack-neutral. The reference interpreter delivers the RESUME signal as a line equal to RESUME on stdin; EOF before that line raises REJECTION.
RESUME Emit a RESUME signal so a peer waiting on a bare HOLD can wake up.
Stack-neutral. The reference interpreter writes the line RESUME to stdout, so two ANAL processes can rendezvous over a pipe.
05 Input / Output

ANAL's I/O model is strictly push-oriented. There is no polling, no watching, no pulling. Either you RECEIVE or you do not.

Command Description
RECEIVE Accept one value from stdin and PUSH it. Blocks until input arrives.
Malformed input raises REJECTION. ANAL is selective about what it RECEIVES.
EXPEL Output the top of stack to stdout. Does not POP.
Non-destructive. ANAL can EXPEL the same value many times without shame.
DISCHARGE Output the top value to stdout and POP it in one motion. Efficient and final.
INGEST Read a file and push its contents as a STRING. Also used for module imports.
INGEST "stdlib/math" — ANAL ingests carefully.
EVACUATE Write the top of stack to a file. Does not POP. Overwrites entirely. needs CONSENT
Requires CONSENT if the file already exists. ANAL does not overwrite without permission.
REQUEST POP a STRING kind and a STRING target; PUSH a BOOL indicating whether the user grants the capability. Recognised kinds: "read", "write", "net". safe in soft mode
In soft mode (the default), pushes TRUE without prompting. In --hard mode, prompts the user on stderr and reads the reply from stdin. See §12 Hard Mode.
RECEIVE_BYTE Read one raw byte from stdin and PUSH it as an INT in the range 0–255. PUSHes -1 on end-of-input.
Use when reading binary or single-character streams. RECEIVE is for prompts; RECEIVE_BYTE is for filters.
EMIT_BYTE POP the top INT and write it to stdout as a single raw byte. Value must be in 0–255 or REJECTION is raised.
Pairs with RECEIVE_BYTE. ANAL does not interpret. ANAL emits.
06 Flow Control

Conditions are values on the stack. Branches consume them. Loops respect them. Nothing is hidden in registers — if it matters, it's on the stack where you can see it.

Command Description
DILATE Open a loop block. Repeats while the top of stack is truthy. POPs condition each iteration.
Must be closed with CONSTRICT. ANAL does not leave things open.
CONSTRICT Close a DILATE block. Re-evaluates top of stack. Truthy → re-enter. Falsy → exit.
IF_TIGHT Branch if the top of stack is truthy. POPs the condition.
IF_LOOSE Branch if the top of stack is falsy. The ELSE of ANAL.
PASSAGE <name>: Declare a named subroutine. Subroutines share the global stack.
ANAL has one stack. No call stack. No local variables. There is only the stack.
ENTER <name> Call a PASSAGE by name.
EXIT Return from a PASSAGE.
ABORT Terminate immediately. Flushes stdout. The stack is abandoned where it stands. 
; Countdown from 5 using DILATE / CONSTRICT
PUSH 5

PROBE
PUSH 0
GT             ; condition: top > 0 ?

DILATE
  DISCHARGE    ; print and pop
  PUSH 1
  SUB
  PROBE
  PUSH 0
  GT           ; re-evaluate for CONSTRICT
CONSTRICT

; Output: 5  4  3  2  1
07 Type System

ANAL is strongly typed and unapologetically so. It will not coerce an integer to a string. It will not guess what you meant. Type conversion is always explicit.

Strict
There is no null type. There is no undefined. A value either exists or the program should not have asked for it. EMPTINESS errors exist for exactly this purpose.
Type Literal Notes
INT 42, -7, 0 64-bit signed integer.
FLOAT 3.14, -0.5 64-bit double. Must contain a decimal point.
STRING "hello" UTF-8. Immutable. Escape sequences: \n \t \\
BOOL TRUE, FALSE Uppercase required. true is a label, not a value.
BLOC [ ... ] A quoted block of code. Square brackets push the enclosed instructions as a single value; the block does not execute until consumed by ENTER, IF_TIGHT, or IF_LOOSE. ANAL's approach to first-class functions.
CAVITY opaque A fixed-size mutable region of INTs, allocated by BUFFER. CAVITY values are first-class — they may be PUSHed, DUPed, passed to a PASSAGE, or stored as a PASSAGE result. They have no literal form; a CAVITY only exists once BUFFER has allocated it. See §9 External Storage.

Arithmetic

All arithmetic ops POP their operands and PUSH a single result. Operand order on the stack: the second-to-top is the left-hand side, the top is the right-hand side. Mixed INT/FLOAT raises REJECTION — convert first.

Command Description
ADD PUSH (a + b). Strings concatenate.
SUB PUSH (a − b). Numeric operands only.
MUL PUSH (a × b). Numeric operands only.
DIV PUSH (a ÷ b). Integer division for INTs; raises REJECTION on divide-by-zero.
MOD PUSH (a mod b). INT operands only.

Comparison

All comparison ops POP two operands and PUSH a BOOL. Operand order matches arithmetic: second-to-top is the left, top is the right.

Command Description
EQ PUSH TRUE if a equals b. Type-strict: 42 and 42.0 are not equal. Convert first.
LT PUSH TRUE if a < b.
GT PUSH TRUE if a > b.
LTE PUSH TRUE if a ≤ b.
GTE PUSH TRUE if a ≥ b.
NOT POP one BOOL, PUSH its negation. The only unary operator in the family.

Conversion

Conversion is always explicit. ANAL will not coerce. Implicit conversion raises REJECTION; you asked for this.

Command Description
TO_INT POP a value, PUSH its INT form. FLOAT truncates toward zero; STRING must parse cleanly or REJECTION.
TO_FLOAT POP a value, PUSH its FLOAT form. INT promotes; STRING must parse cleanly.
TO_STRING POP a value, PUSH its STRING form. Always succeeds.
08 String Inspection

STRING values in ANAL are immutable. They cannot be modified in place. They can, however, be inspected: their length, their bytes, and their substrings are all retrievable. Inspection is non-destructive and requires no ceremony.

Note
Indices are zero-based and count bytes, not codepoints. ANAL strings are UTF-8; a STRING containing multi-byte characters reports its byte length, not its glyph count. ANAL counts what it can prove.
Command Description
STRLEN POP a STRING, PUSH its length in bytes as an INT.
Non-destructive in the consent sense; the STRING itself is consumed by the POP, as with any other op. DUP first to keep it.
CHARAT POP an INT index and a STRING, PUSH the byte at that index as an INT in 0–255. Out-of-bounds raises CAVITY_BREACH.
Stack order: STRING below, index on top. "hello" 0 CHARAT pushes 104.
SUBSTR POP an INT length, an INT start, and a STRING; PUSH the substring of length bytes beginning at start.
Stack order from bottom to top: STRING, start, length. Out-of-bounds at either end raises CAVITY_BREACH. A length of 0 PUSHes the empty STRING; ANAL accepts emptiness when it was asked for.
09 External Storage

The stack is ANAL's primary memory. For programs that require random access to indexed data — interpreters, parsers, simulations — ANAL provides the CAVITY: a fixed-size, mutable, INT-valued region allocated by BUFFER. CAVITY semantics extend ANAL's existing consent discipline to a new region. Reads are free; writes are ceremonial.

Strict
A CAVITY is fixed in size at allocation and cannot grow. Writing past its end raises CAVITY_BREACH. ANAL does not stretch what it has already shaped.

Allocation and Access

Command Description
BUFFER <n> Allocate a new CAVITY of n INT cells, all initialised to 0, and PUSH it. n is a compile-time literal INT > 0. A non-positive literal raises HOLLOW at compile time.
Allocation itself is non-destructive — nothing is overwritten. A CAVITY freshly returned by BUFFER is hollow but not uninitialised.
BUFFER Bare form: POP an INT from the stack and use it as the cell count, otherwise behaving exactly as BUFFER <n>. A non-positive popped value raises HOLLOW at runtime.
Same op, two adicities — matches the HOLD precedent. Use the literal form when the size is fixed by the program, the bare form when it came from input. ANAL accepts what it was given, including how it was given.
BUFGET POP an INT index, read the cell at that index in the CAVITY one position below it, and PUSH the result. The CAVITY remains on the stack — only the index is consumed. Out-of-bounds raises CAVITY_BREACH.
Stack order: CAVITY below, index on top. After the op: CAVITY stays, the read value sits above it.
BUFSET POP an INT value and an INT index, write the value at the indexed cell of the CAVITY one position below. The CAVITY remains on the stack — only the index and value are consumed. needs PREP and CONSENT
Stack order from bottom to top: CAVITY, index, value. The same ceremony as INSERT: PREP arms the write; CONSENT permits it. Forgetting either raises TIGHTNESS or REFUSAL, respectively.
BUFLEN Read the cell count of the CAVITY at the top of the stack and PUSH it as an INT. The CAVITY remains on the stack.
Inspection, not consumption. The CAVITY is a persistent region; reading its size does not disturb it.

Compile-time-indexed access

For the common case where a CAVITY is used as a fixed set of named cells (a "locals frame," in the established idiom), the index is known at the source. LOAD and STORE take the index as a literal, freeing the stack from the bookkeeping that BUFGET and BUFSET require.

Command Description
LOAD <i> Read cell i of the CAVITY at the top of the stack and PUSH the result as an INT. The CAVITY remains. Out-of-bounds raises CAVITY_BREACH at runtime.
i is a non-negative compile-time literal. For a runtime-computed index, use BUFGET.
STORE <i> POP an INT value and write it to cell i of the CAVITY now on top of the stack. The CAVITY remains. needs PREP and CONSENT
Same ceremony as BUFSET: forgetting either latch raises TIGHTNESS or REFUSAL. i is a non-negative compile-time literal. For a runtime-computed index, use BUFSET.

Persistence and Consumption

A CAVITY is a persistent region. The ops that act on it (BUFGET, BUFSET, BUFLEN, LOAD, STORE) do not consume the CAVITY itself; they consume only the indices and values they were given. This is the same pattern that governs every other persistent destination in ANAL: EXPEL does not consume stdout, EVACUATE does not consume the file, INSERT does not consume the stack. Acts performed on an enduring thing do not destroy the thing.

Discarding a CAVITY is an explicit act. When you are finished with one, POP it. ANAL does not garbage-collect attention.

Consent Semantics

A BUFSET or STORE consumes one PREP and one CONSENT, exactly as a stack INSERT would. A program that performs many writes in succession must arm the ceremony for each one. This is not an oversight. ANAL considers a sequence of unwitnessed writes the precise condition the consent state machine was designed to prevent. LOAD, BUFGET, and BUFLEN require no ceremony: reading is not violation.

The write ceremony is local to the CAVITY: a CONSENT armed before a stack INSERT does not authorise a subsequent BUFSET or STORE, and vice versa. Each region demands its own.

On verbosity
The ceremony of PREP CONSENT STORE <i> is intentional. Programs that perform many writes are expected to wrap the operation in a PASSAGE — the ceremony lives in the definition; the call sites read clean. ANAL externalises ceremony to where it belongs; it does not eliminate it. A language that makes destruction cheap is the precise thing ANAL was written against.
Note
A CAVITY participates in the type checker. BUFGET on a STRING raises MISMATCH at probe time. A PASSAGE that returns a CAVITY declares it the same way it declares any other return: by leaving it on the stack at EXIT.
v0.4
The consent ceremony — PREP, CONSENT, CLENCH — is itself checked statically. A BUFSET or STORE on a path that does not arm both latches fails at probe time, with the same TIGHTNESS or REFUSAL the runtime would have raised. See §10 Static Checking for the full treatment.
10 Static Checking

Every program runs through a static check between parse and execute. The check is called probing — invoked directly with anal probe <file>, and implicitly by every anal run. A program that fails the probe never reaches the VM.

The probe tracks two things in parallel: an abstract stack, and an abstract consent effect. They are independent. A program may be well-typed and still fail the effect check; it may satisfy the effect check and still fail the type check. Both must pass.

The abstract stack

At each program point the checker holds an ordered sequence of types — one slot per value the runtime would have. The slot types are INT, FLOAT, STRING, BOOL, BLOC, CAVITY, and <any>. <any> arises at control-flow merges where two arms disagree on the type of a slot; it is matchable against any concrete type, and ops with a precise type rule on that slot will still execute under it.

Each op carries a typing rule. ADD on two INTs yields an INT; on two FLOATs, a FLOAT; on two STRINGs, concatenation. Any other pairing raises MISMATCH. EXTRACT shrinks the stack by one; INSERT grows it by one; FLUSH empties it. PASSAGE bodies are re-checked at every call site against the caller's actual stack, which gives ad-hoc polymorphism without generics — PASSAGE square: DUP MUL EXIT type-checks on both INTs and FLOATs because each call site walks the body afresh.

The consent effect

Alongside the abstract stack, the checker walks a three-field effect: the state of PREP, the state of CONSENT, and the current CLENCH depth. Each latch is one of three values — Unarmed, Armed, or Top. Top is the join of disagreeing branches: armed on one path, unarmed on another, neither provable.

A destructive op — INSERT, EXTRACT, FLUSH, BUFSET, STORE — requires its precondition latch to be Armed on every path that reaches it. Both Unarmed and Top are rejected: the checker insists on a positive proof of arming, not the absence of evidence against it.

  • INSERT without armed PREP raises TIGHTNESS (E001) at probe time.
  • EXTRACT, FLUSH without armed CONSENT raise REFUSAL (E004) at probe time.
  • BUFSET, STORE require both; missing either raises the corresponding error.
  • Any destructive op inside a CLENCH raises LOCKDOWN (E005), as does any op that cannot prove the stack is unclenched on every reaching path.

The same error codes the runtime emits are emitted by the checker; the difference is only where the failure is detected. A program that probes clean cannot raise TIGHTNESS or REFUSAL at runtime, and cannot raise LOCKDOWN in any case the checker was able to decide.

Joins at control flow

At an IF_TIGHT or IF_LOOSE merge, the checker walks the arm body on copies of both the stack and the effect, then joins the result back into the outer state. Two latches in the same state stay in that state; latches that disagree raise to Top. The CLENCH depth is preserved when both paths agree on the depth, and collapses to "unknown" when they do not.

A DILATE loop is checked the same way, with the entry effect joined against the body-exit effect — the body may run zero times or many. Arming that the body never disturbs survives the loop; arming the body clears, even once, does not.

The consequence is precise but conservative. Arming the same latch on both arms of an IF_TIGHT survives the merge:

PREP
PUSH cond
IF_LOOSE [
  RELAX
  PREP
]
INSERT 1 99  ; legal — PREP armed on every path

Arming on one path and not the other does not:

PUSH cond
IF_TIGHT [ PREP ]
INSERT 1 99  ; TIGHTNESS at probe time

What the checker does not prove

The checker is about types and arming, not values. PUSH 0 DIV is well-typed; it raises REJECTION at runtime when the divisor is read. A RECEIVE followed by "yes" EQ may or may not produce TRUE — that is for the user, not the checker. And EVACUATE's consent requirement is conditional on whether the target file exists, which the checker cannot know; that check stays at runtime.

The runtime checks for TIGHTNESS, REFUSAL, LOCKDOWN, and PREMATURE_RELEASE remain in place. They protect against the cases the checker is too coarse to catch — dynamic BLOC bodies, conditional file overwrites, and any future op whose preconditions depend on runtime values. A program that fails the probe never runs; a program that probes clean still runs through the same runtime guards.

11 The Ledger

The runtime checks consent. It does not, by default, remember that it asked. With --ledger, every destructive operation a program performs is appended to a hash-chained binary file — the .sphlog. anal audit reads it back and verifies that the recorded operations match the source they were recorded against and that no entry has been edited since.

The ledger is optional. Programs that do not opt in run exactly as before. Programs that do opt in pay one fixed-size append per destructive op.

Recording a run

$ anal run --ledger run.sphlog program.anal

The named file must not already exist; anal refuses to overwrite a ledger, since a single ledger pairs one-to-one with a single run. The header is written immediately and includes a blake3 hash of the source file. Each subsequent destructive op (INSERT, EXTRACT, FLUSH, BUFSET, STORE, and an EVACUATE that overwrote an existing file) appends one fixed-size record. Failed ops do not appear in the log — the ledger records what was done, not what was attempted.

Verbose mode

$ anal run --ledger run.sphlog --ledger-verbose program.anal

By default the ledger records only consummated destructive acts. With --ledger-verbose, the five arming ops — PREP, CONSENT, CLENCH, RELEASE, RELAX — are also recorded, in execution order. The result is a complete consent trail: not just what was done but what was prepared, how long it sat armed, and whether anything was armed and then released without being spent.

Records share the same fixed-size layout regardless of verbosity; the only difference is which op_tag values appear. A verbose ledger increases the file's version field to 0x02, so a reader from an earlier release rejects it with UnsupportedVersion instead of silently misreading.

Auditing a run

$ anal audit run.sphlog program.anal
✓ run.sphlog — 4 record(s) verified against program.anal.

The audit performs three checks, in order, and fails fast on the first that does not pass:

  • Source identity. The ledger's recorded source_hash must equal the blake3 of the supplied source. A mismatch raises LEDGER_DRIFT (E017) — the two files do not belong together.
  • Chain integrity. Each record carries the prior record's hash. The audit walks the chain; a hash that does not match the prior record raises BROKEN_CHAIN (E016). An editor who changes one past record invalidates every subsequent prev_hash.
  • Source agreement. Each record names an op and a byte span in the source. The audit compiles the source, locates the op at that span, and confirms it matches. A mismatch raises LEDGER_GAP (E018) — the source has changed since the ledger was recorded.

Examining a verified ledger

By default anal audit only prints a one-line verification result. Pass --examine to additionally print every verified record as an aligned table:

$ anal audit --examine run.sphlog program.anal
✓ run.sphlog — 4 record(s) verified against program.anal.

  seq  timestamp                    op       location               stack
  ───  ───────────────────────────  ───────  ─────────────────────  ─────────────────────────────────
  0    2026-05-25T16:54:01.098644Z  INSERT   ledger_demo.anal:27:1  [INT, INT, INT]
  1    2026-05-25T16:54:01.098649Z  INSERT   ledger_demo.anal:32:1  [INT, INT, INT, INT]
  2    2026-05-25T16:54:01.098653Z  EXTRACT  ledger_demo.anal:37:1  [INT, INT, INT, INT, ...] depth=5
  3    2026-05-25T16:54:01.098688Z  FLUSH    ledger_demo.anal:47:1  [INT, INT, INT, INT]

Columns are: the sequence number, the UTC timestamp the op fired (microsecond precision), the op name, the source file with the 1-based line and column of the recorded span, and the pre-op stack shape. The stack column lists up to four top-of-stack types in top-first order; when the actual depth exceeded four, an ellipsis and a depth=N suffix appear. --examine is read-only — verification still fails fast on a chain break or source drift, and no table is printed in that case.

.sphlog file format

All multi-byte integers are little-endian. The format is fixed-shape, append-only, and self-describing.

Header (40 bytes, once)

offset  size  field         notes
  0      4    magic         b"SPHL"
  4      1    version       0x02 (bumped in v0.4.4 to admit arming-op tags)
  5      3    reserved      zero
  8     32    source_hash   blake3 of source file at run time

Record (72 bytes, repeated)

offset  size  field         notes
  0      8    seq           u64, 0-based
  8      8    ts_micros     i64, unix epoch microseconds
 16      1    op_tag        destructive: 1=INSERT 2=EXTRACT 3=FLUSH
                                         4=BUFSET 5=STORE   6=EVACUATE
                            arming (verbose only):
                                         7=PREP   8=CONSENT 9=CLENCH
                                         10=RELEASE 11=RELAX
 17      1    reserved      zero
 18      4    span_start    u32, byte offset into source
 22      4    span_end      u32, byte offset into source
 26      4    stack_depth   u32, depth *before* the op fired
 30      1    top_n         0..=4, slots of stack types recorded
 31      4    top_types     first top_n are valid:
                            1=INT 2=FLOAT 3=STRING
                            4=BOOL 5=BLOC 6=CAVITY
 35      5    reserved      zero
 40     32    prev_hash     this_hash of the prior record;
                            zeros for the genesis record

The this_hash of each record is blake3 over bytes 0..72 — including the prev_hash just written. It is not stored on disk; readers recompute it. This is what makes the chain self-validating: any byte changed in record n changes the hash that record n+1 expected, and the audit halts at record n+1.

There is no record count in the header. Readers iterate until EOF. A torn write that leaves a partial record at the tail is detected and reported as RecordTruncated.

What is recorded, and what is not

Each record holds an op, a source span, a pre-op stack depth, and up to four top-of-stack type tags. Values are not recorded. The ledger captures the shape and the act, never the contents — a ledger from a production run carries no payload data, so it is safe to ship alongside a bug report or store with the source.

Arming operations — PREP, CONSENT, CLENCH, RELEASE, RELAX — are not logged. The ledger records consummated acts, not preparations. A program that arms and then aborts before acting leaves no audit trail; this is correct, because no destruction occurred.

An EVACUATE that creates a new file is not logged either. Creation is benign; the ledger tracks consent-bearing acts.

On signing
The hash chain detects edits, not impersonation. A v0.5 follow-up will add Ed25519 signatures keyed by an operator identity; for now, a .sphlog proves only that its contents have not been altered since they were written, not that any particular party wrote them. For local-machine audits this is sufficient. For shared infrastructure, treat the ledger as authenticated by the medium it travelled on (signed releases, attested CI runs) until v0.5 lands.
12 Hard Mode

The interpreter, by default, runs with ambient filesystem authority: any program can INGEST any readable file and EVACUATE to any writable path the operating system permits. This matches the behaviour of every scripting language since shell. It is not what every program wants.

Hard mode, engaged with anal run --hard file.anal, strips that authority. In hard mode there is no ambient access to the filesystem. INGEST and EVACUATE raise OUTSIDE (E019) unless the program has explicitly asked for, and the user has explicitly granted, the exact target.

Asking is done with a new op: REQUEST.

REQUEST

Stack effect: ( target:STRING kind:STRING -- granted:BOOL ).

Three kinds are recognised: "read", "write", "net". Unknown kinds raise REJECTION at runtime — the program is asking for something the runtime cannot enforce. (The "net" kind parses and grants today but no op consumes it yet; network primitives are a planned follow-up.)

In soft mode (the default), REQUEST trivially pushes TRUE. No prompt fires, no stderr is touched. A program written for hard mode runs identically under soft mode — it simply never sees a denial.

In hard mode, REQUEST writes a two-line prompt to stderr and reads one line of reply from stdin:

ANAL requests capability: READ "config.txt"
  Grant? (yes / no / always)

Recognised replies, case-insensitive:

  • yes / y — grant once. The capability is consumed by the next matching I/O op only.
  • always / a — grant for the rest of the run. The next REQUEST for the same exact (kind, target) pair will not prompt and returns TRUE immediately.
  • no, anything else, or EOF — push FALSE. The program decides what to do; the runtime does not abort on a denial.

What the gates protect

Under hard mode, two ops consult the granted set before doing their work:

  • INGEST <path> requires the path in the read set. Failure raises OUTSIDE (E019) before the file is touched.
  • EVACUATE <path> requires the path in the write set. Failure raises OUTSIDE before any disk activity. The existing CONSENT requirement for overwriting an existing file still applies on top of the capability check.

Other I/O — DISCHARGE, EXPEL, RECEIVE, EMIT_BYTE, RECEIVE_BYTE — talks to the terminal that launched the program and is not gated. Writing to the terminal that asked is not reaching outside.

Path matching

Matches are exact and string-literal. REQUEST "data/" "read" grants data/ and nothing else; it does not grant data/file.txt or data/sub/. A program that needs to read several files must request each by name. This is deliberate — the point of --hard is that broad authority is not the default.

A future RFC may add pattern matching (glob, prefix) if the deny-by-default ergonomics prove too tight in practice. For now, exact-match is the only form.

Composing with the ledger

A run can use both --hard and --ledger at once. The ledger records destructive ops as it always does; the capability layer prevents I/O against ungranted targets independently. REQUEST outcomes are not logged to the ledger — the ledger records consummated destructive acts, not capability decisions. (A separate capability ledger may land in v0.5.)

Note
Hard mode is interactive by design: the user is in the loop on every capability decision. There is no flag to pre-grant capabilities from the command line — that would defeat the point. Unattended use (CI, batch processing) will get a separate --grants <file> mechanism in v0.5; the design is deliberately deferred until the interactive shape has time to settle.

15 Contributing Policy
Policy
ANAL does not accept pull requests. This is not a bug. All changes flow in one direction. There is no pulling. This models good data hygiene: data arrives, you process it, it exits. You do not go looking for it.

The ANAL project uses Push-Oriented Development (POD). Patches are submitted by email. Issues are raised by insertion into the tracker. The RFC PASSAGE process governs all significant changes.

Permitted
  • Submitting patches by email
  • Raising issues (INSERTION into tracker)
  • Forking for personal use (with CONSENT)
  • Reading the source (PROBE always allowed)
  • Proposing via RFC PASSAGE process
  • Sponsoring the project
Forbidden
  • Pull requests of any kind
  • Proposing a threading model
  • Adding null to the type system
  • Lowercase keywords in any example
  • Implicit type coercions in core
  • Renaming PENETRATION_DEPTH

The RFC PASSAGE Process

Significant changes must go through the RFC (Request For Consideration) PASSAGE process. An RFC describes the proposed change, its motivation, its impact on existing programs, and a precise specification. RFCs are PUSHed to the RFC repository. The core team PROBEs each RFC, deliberates, and either INSERTs it into the roadmap or EXTRACTs it with a written explanation. CONSENT of a majority of core maintainers is required for acceptance.

The RFC PASSAGE process typically takes 6–18 months. ANAL does not rush. Haste causes TIGHTNESS errors.

17 FAQ
Why can't I use pull requests?
Because ANAL does not pull. This is a design decision, not an oversight. If you need to pull, you are using the wrong language. ANAL accepts PUSHes only. Data arrives, you process it, it exits. You do not go looking for it.
Why is there no null type?
The ANAL design committee considered null for approximately eleven minutes before deciding that a value either exists or the question should not have been asked. EMPTINESS errors tell you when you have overstepped. They are considered a feature.
Why is CONSENT a keyword and not just a flag?
Because consent should be explicit, visible, and in the code. A flag hidden in a config file is not consent. ANAL requires that you write the word, in the program, before you do something that cannot be undone. This is not philosophical. It is engineering.
Can I add threads?
No. The ANAL issue tracker has 47 closed issues on this topic. The answer is no. It has always been no. It will always be no. ANAL is sequential. This is a load-bearing architectural decision. Please stop asking.
Why is the package manager called suppository?
Because packages are inserted directly. There is no other name that would be more accurate. The team considered alternatives. There were none.
Why is the compiled bytecode format called .sph?
Short for Sphincter Bytecode. The sphincter is the body's most reliable one-way valve. We found this appropriate for a language that controls exactly what enters and what exits. The compiler team was very pleased with this decision.
Is ANAL production-ready?
ANAL v1.0 is considered stable. The specification is frozen. Several organisations use ANAL in production and prefer not to say so publicly. We respect this.
Is ANAL Turing-complete?
Yes. Runtime-sized BUFFER, indexed BUFGET/BUFSET, and unbounded DILATE loops cover all three requirements. The Collatz interpreter at examples/collatz.anal records its trajectory into a CAVITY whose size came from the input — the requirement for strict completeness. Bounded earlier; strict since v0.3.
