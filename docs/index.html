Language Specification · Version 1.0.0
ANAL

Append-oriented, Narrow-access Language. A strictly sequential, stack-based programming language designed around one governing principle: everything enters from one end, in order, with consent.

stack-based strongly typed no pull requests consent enforced turing complete* .anal · .sph
Note
This document describes ANAL v1.0. The current reference interpreter implements v0.2 — a strict subset. See the milestones for what's shipped and what's still spec'd.
01 Design Philosophy
Principle
"ANAL does not meet you halfway. ANAL expects you to come to it." — The ANAL Manifesto, §1

ANAL was designed around a single governing principle: everything enters from one end, in order, with consent. There are no shortcuts, no backdoors, no random access. The stack is sacred.

Unlike languages that permit arbitrary memory access, object mutation, or concurrent execution, ANAL enforces sequentiality as a virtue, not a limitation. If you need random access, you have chosen the wrong language. ANAL is not sorry.

Core Tenets

Append only, by default. The natural direction of ANAL is forward. Removal is possible but deliberate.

Consent before destruction. Any operation that permanently removes or modifies data at depth requires the CONSENT flag. This is not optional.

Preparation before penetration. Deep stack operations require PREP. Skipping it raises a TIGHTNESS error. The runtime does not apologise for this.

No pulling. ANAL does not support pull requests, pull-based I/O, or any metaphor involving pulling. Everything is pushed.

Sequential or nothing. ANAL has no threading model. Proposals to add one are closed without comment.

02 Syntax & Structure

ANAL programs are plain text files with the .anal extension. Each statement occupies one line. Comments begin with ;. ANAL is case-sensitive — all keywords are uppercase. Writing a keyword in lowercase raises a CASING error.

; This is a comment.

PUSH 42         ; push integer
PUSH "hello"     ; push string
PUSH 3.14       ; push float

loop_start:     ; label declaration
  DILATE        ; begin loop block
    EXPEL
  CONSTRICT     ; end loop block

File Structure

An ANAL program has three optional sections, in strict order: header, definitions, execution.

; 1. Header — metadata and imports
ANAL "my_program" VERSION 1
INGEST "stdlib/math"

; 2. Definitions — PASSAGES (functions)
PASSAGE square:
  DUP
  MUL
EXIT

; 3. Execution — the main body
PUSH 9
ENTER square
EXPEL    ; → 81
03 Stack Operations

The stack is the heart of ANAL. All data lives on the stack. All computation moves through the stack. There is only the stack.

Command Args Description
PUSH <value> Insert a value at the top of the stack. The primary method of input. ANAL strongly prefers PUSH to all alternatives.
Accepts INT, FLOAT, STRING, BOOL, BLOC. Does not accept nulls. ANAL has no concept of nothing.
POP Remove and return the top value. The value is gone.
Raises EMPTINESS if stack is empty. Raises LOCKDOWN if CLENCHed.
PROBE Inspect the top value without removal. Non-invasive. PROBE is the only truly safe operation in ANAL. always safe
May be called at any time, in any state, even during CLENCH.
INSERT <depth> <val> Inject a value at an arbitrary depth. Requires PREP immediately prior. needs PREP
Raises TIGHTNESS if PREP was not called. Raises PENETRATION_DEPTH if depth exceeds stack size.
EXTRACT <depth> Remove a value from an arbitrary depth. The gap collapses cleanly. needs CONSENT
Raises REFUSAL without CONSENT. Raises PENETRATION_DEPTH if out of bounds.
SWAP Exchange the top two values.
DUP Duplicate the top value and push the copy.
DEPTH Push the current stack depth as an INT. Useful before INSERT or EXTRACT.
FLUSH Clear the entire stack. Cathartic. Permanent. needs CONSENT
Raises REFUSAL without prior CONSENT.
04 Control & State
Command Args Description
PREP Prepare the stack for deep operations. Sets internal readiness flag. Must immediately precede INSERT.
Readiness flag clears after one INSERT. Good hygiene. Always PREP.
CLENCH Freeze the stack. No write operations permitted until RELEASE. PROBE still works.
Nested CLENCHes are cumulative. Each requires a matching RELEASE.
RELEASE Unfreeze the stack. Decrements CLENCH depth by one.
Raises PREMATURE_RELEASE if called on an unclenched stack.
CONSENT Set the consent flag. Required before FLUSH, EXTRACT, or any destructive deep operation. mandatory
Clears automatically after one destructive operation. Must be re-given each time. ANAL does not assume ongoing consent.
RELAX Clear any armed PREP or CONSENT flag. always safe
Idempotent: a no-op if neither flag is armed. Allowed during CLENCH since it does not touch the stack. Consent withdrawn before use is consent never given. ANAL forgives forgetting.
EXPAND <n> Reserve capacity for n more values above the current depth. Call before you need it, not after.
Monotonic: subsequent EXPANDs only raise the cap, never lower it. A push past the cap raises OVERFLOW. Stacks with no EXPAND are uncapped and never overflow.
HOLD [ms] Pause execution. With an INT operand, sleeps that many milliseconds. Without one, waits for a RESUME signal.
Stack-neutral. The reference interpreter delivers the RESUME signal as a line equal to RESUME on stdin; EOF before that line raises REJECTION.
RESUME Emit a RESUME signal so a peer waiting on a bare HOLD can wake up.
Stack-neutral. The reference interpreter writes the line RESUME to stdout, so two ANAL processes can rendezvous over a pipe.
05 Input / Output

ANAL's I/O model is strictly push-oriented. There is no polling, no watching, no pulling. Either you RECEIVE or you do not.

Command Description
RECEIVE Accept one value from stdin and PUSH it. Blocks until input arrives.
Malformed input raises REJECTION. ANAL is selective about what it RECEIVES.
EXPEL Output the top of stack to stdout. Does not POP.
Non-destructive. ANAL can EXPEL the same value many times without shame.
DISCHARGE Output the top value to stdout and POP it in one motion. Efficient and final.
INGEST Read a file and push its contents as a STRING. Also used for module imports.
INGEST "stdlib/math" — ANAL ingests carefully.
EVACUATE Write the top of stack to a file. Does not POP. Overwrites entirely. needs CONSENT
Requires CONSENT if the file already exists. ANAL does not overwrite without permission.
06 Flow Control

Conditions are values on the stack. Branches consume them. Loops respect them. Nothing is hidden in registers — if it matters, it's on the stack where you can see it.

Command Description
DILATE Open a loop block. Repeats while the top of stack is truthy. POPs condition each iteration.
Must be closed with CONSTRICT. ANAL does not leave things open.
CONSTRICT Close a DILATE block. Re-evaluates top of stack. Truthy → re-enter. Falsy → exit.
IF_TIGHT Branch if the top of stack is truthy. POPs the condition.
IF_LOOSE Branch if the top of stack is falsy. The ELSE of ANAL.
PASSAGE <name>: Declare a named subroutine. Subroutines share the global stack.
ANAL has one stack. No call stack. No local variables. There is only the stack.
ENTER <name> Call a PASSAGE by name.
EXIT Return from a PASSAGE.
ABORT Terminate immediately. Flushes stdout. The stack is abandoned where it stands. 
; Countdown from 5 using DILATE / CONSTRICT
PUSH 5

PROBE
PUSH 0
GT             ; condition: top > 0 ?

DILATE
  DISCHARGE    ; print and pop
  PUSH 1
  SUB
  PROBE
  PUSH 0
  GT           ; re-evaluate for CONSTRICT
CONSTRICT

; Output: 5  4  3  2  1
07 Type System

ANAL is strongly typed and unapologetically so. It will not coerce an integer to a string. It will not guess what you meant. Type conversion is always explicit.

Strict
There is no null type. There is no undefined. A value either exists or the program should not have asked for it. EMPTINESS errors exist for exactly this purpose.
Type Literal Notes
INT 42, -7, 0 64-bit signed integer.
FLOAT 3.14, -0.5 64-bit double. Must contain a decimal point.
STRING "hello" UTF-8. Immutable. Escape sequences: \n \t \\
BOOL TRUE, FALSE Uppercase required. true is a label, not a value.
BLOC [ ... ] A quoted block of code. Square brackets push the enclosed instructions as a single value; the block does not execute until consumed by ENTER, IF_TIGHT, or IF_LOOSE. ANAL's approach to first-class functions.

Arithmetic

All arithmetic ops POP their operands and PUSH a single result. Operand order on the stack: the second-to-top is the left-hand side, the top is the right-hand side. Mixed INT/FLOAT raises REJECTION — convert first.

Command Description
ADD PUSH (a + b). Strings concatenate.
SUB PUSH (a − b). Numeric operands only.
MUL PUSH (a × b). Numeric operands only.
DIV PUSH (a ÷ b). Integer division for INTs; raises REJECTION on divide-by-zero.
MOD PUSH (a mod b). INT operands only.

Comparison

All comparison ops POP two operands and PUSH a BOOL. Operand order matches arithmetic: second-to-top is the left, top is the right.

Command Description
EQ PUSH TRUE if a equals b. Type-strict: 42 and 42.0 are not equal. Convert first.
LT PUSH TRUE if a < b.
GT PUSH TRUE if a > b.
LTE PUSH TRUE if a ≤ b.
GTE PUSH TRUE if a ≥ b.
NOT POP one BOOL, PUSH its negation. The only unary operator in the family.

Conversion

Conversion is always explicit. ANAL will not coerce. Implicit conversion raises REJECTION; you asked for this.

Command Description
TO_INT POP a value, PUSH its INT form. FLOAT truncates toward zero; STRING must parse cleanly or REJECTION.
TO_FLOAT POP a value, PUSH its FLOAT form. INT promotes; STRING must parse cleanly.
TO_STRING POP a value, PUSH its STRING form. Always succeeds.
08 Error Reference

ANAL errors are descriptive, unambiguous, and delivered without sympathy. The runtime considers excessive hand-holding a design flaw.

TIGHTNESS
E001
INSERT attempted without prior PREP. Always prepare.
EMPTINESS
E002
POP, PROBE, or DISCHARGE on an empty stack.
OVERFLOW
E003
Stack capacity exceeded. Use EXPAND proactively.
REFUSAL
E004
Destructive operation attempted without CONSENT.
LOCKDOWN
E005
Write attempted on a CLENCHed stack. RELEASE first.
PENETRATION_DEPTH
E006
INSERT or EXTRACT depth exceeds current stack size.
REJECTION
E007
Type mismatch or malformed input. ANAL rejects what does not belong.
PREMATURE_RELEASE
E008
RELEASE called when stack is not CLENCHed. Unseemly.
CASING
E009
Keyword written in lowercase. ANAL will not stoop.
FOREIGN_BODY
E010
Unrecognised token. ANAL does not accept things it does not know.
RUPTURE
E011
Unclosed DILATE block. ANAL does not leave things open.
PASSAGE_NOT_FOUND
E012
ENTER called on an undefined PASSAGE name.
MISMATCH
E013
Static type or stack-shape disagreement, caught at probe time. ANAL refuses to run what it has not first understood.
09 suppository
💊
suppository
The official ANAL package manager. Dependencies inserted directly — no pulling required.

Packages are hosted on the Sphincter Registry at registry.sph — a curated repository of ANAL-compatible libraries. All packages are verified for type safety before admission. Loose types are rejected at the gate.

$ suppository insert anal-math
Probing registry.sph for anal-math...
Preparing insertion point...
✓ anal-math@2.1.0 inserted at depth 0

$ suppository extract anal-math --consent
✓ anal-math extracted cleanly. No residue.

$ suppository flush
⚠ Destructive operation. --consent flag required.

$ suppository flush --consent
✓ All packages flushed. Stack is clean.

The .sphfile

Project dependencies are declared in a .sphfile in the project root.

; .sphfile — project dependency manifest

project:      "my_program"
version:      "1.0.0"
anal_version: ">=1.0.0"

dependencies:
  anal-math      "^2.0.0"
  anal-string    "^1.4.0"
  anal-io        "^3.0.0"

dev_dependencies:
  anal-test      "^1.0.0"   ; testing framework
  anal-probe     "^0.9.0"   ; stack inspector / debugger

suppository Commands

CommandDescription
insert <pkg> Insert a package from the Sphincter Registry.
extract <pkg> --consent Remove a package. Requires --consent. Cannot be undone.
probe List all currently inserted packages and versions. Non-destructive.
flush --consent Remove all packages. Full dependency flush. Do not do this casually.
prep Prepare the insertion environment from .sphfile. Equivalent to npm install.
publish Publish to the Sphincter Registry. Your package will be reviewed for tightness before acceptance.
clench Lock dependency versions. Writes a .sph.lock file.

10 Contributing Policy
Policy
ANAL does not accept pull requests. This is not a bug. All changes flow in one direction. There is no pulling. This models good data hygiene: data arrives, you process it, it exits. You do not go looking for it.

The ANAL project uses Push-Oriented Development (POD). Patches are submitted by email. Issues are raised by insertion into the tracker. The RFC PASSAGE process governs all significant changes.

Permitted
  • Submitting patches by email
  • Raising issues (INSERTION into tracker)
  • Forking for personal use (with CONSENT)
  • Reading the source (PROBE always allowed)
  • Proposing via RFC PASSAGE process
  • Sponsoring the project
Forbidden
  • Pull requests of any kind
  • Proposing a threading model
  • Adding null to the type system
  • Lowercase keywords in any example
  • Implicit type coercions in core
  • Renaming PENETRATION_DEPTH

The RFC PASSAGE Process

Significant changes must go through the RFC (Request For Consideration) PASSAGE process. An RFC describes the proposed change, its motivation, its impact on existing programs, and a precise specification. RFCs are PUSHed to the RFC repository. The core team PROBEs each RFC, deliberates, and either INSERTs it into the roadmap or EXTRACTs it with a written explanation. CONSENT of a majority of core maintainers is required for acceptance.

The RFC PASSAGE process typically takes 6–18 months. ANAL does not rush. Haste causes TIGHTNESS errors.

12 FAQ
Why can't I use pull requests?
Because ANAL does not pull. This is a design decision, not an oversight. If you need to pull, you are using the wrong language. ANAL accepts PUSHes only. Data arrives, you process it, it exits. You do not go looking for it.
Why is there no null type?
The ANAL design committee considered null for approximately eleven minutes before deciding that a value either exists or the question should not have been asked. EMPTINESS errors tell you when you have overstepped. They are considered a feature.
Why is CONSENT a keyword and not just a flag?
Because consent should be explicit, visible, and in the code. A flag hidden in a config file is not consent. ANAL requires that you write the word, in the program, before you do something that cannot be undone. This is not philosophical. It is engineering.
Can I add threads?
No. The ANAL issue tracker has 47 closed issues on this topic. The answer is no. It has always been no. It will always be no. ANAL is sequential. This is a load-bearing architectural decision. Please stop asking.
Why is the package manager called suppository?
Because packages are inserted directly. There is no other name that would be more accurate. The team considered alternatives. There were none.
Why is the compiled bytecode format called .sph?
Short for Sphincter Bytecode. The sphincter is the body's most reliable one-way valve. We found this appropriate for a language that controls exactly what enters and what exits. The compiler team was very pleased with this decision.
Is ANAL production-ready?
ANAL v1.0 is considered stable. The specification is frozen. Several organisations use ANAL in production and prefer not to say so publicly. We respect this.
